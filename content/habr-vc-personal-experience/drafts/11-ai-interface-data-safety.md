# Как я перестал путать вайб-кодинг с учётной системой

Варианты заголовка:

- Что вайб-кодинг не закрывает: попытка внедрить ИИ-сгенерированное приложение в реальную работу
- AI нарисовал мне приложение за вечер. Потом начались вопросы про данные и права
- Вайб-кодинг хорош для интерфейса, но не должен владеть данными
- ИИ делает интерфейс. Учётная система начинается там, где появляются права и история

## TL;DR

Вайб-кодинг (генерация приложений через ИИ из текстового описания) красиво рисует первый экран и плохо закрывает то, что делает программу учётной системой: права доступа на стороне сервера, целостность данных при конкурентных правках, аудит, бэкап, экспорт, поддержку через год. На практике после «демо за вечер» команда тратит 2–3 месяца на переписывание под реальную работу — то есть «скорость ИИ» уходит на закрытие архитектурных вопросов, которые он сам и создал.

В статье — конкретный сценарий (учёт сервисных заявок), список вопросов, на которых ИИ-сгенерированное приложение ломается, и предложение, где правильно расположить шов: ИИ-генератор — на интерфейсе, платформа — на данных.

---

## Откуда я смотрю на эту тему

Я люблю момент, когда из короткого описания задачи появляется рабочий экран: форма, таблица, кнопки, фильтры. Но в одном из внутренних сценариев я слишком быстро решил, что красивое демо уже похоже на систему. Через неделю стало ясно: ИИ делает интерфейс, а данные, права и история должны жить в платформе.

Веду продукт, который последние полгода предлагали заменить вайб-кодингом в десятке внедрений: «зачем нам ваш конструктор, мы за вечер сделаем то же самое в Lovable/Bolt/v0». На большой выборке наблюдений сложилась картина, чем заканчиваются такие попытки и где именно вайб-кодинг даёт сбой. Это не критика инструментов — они хороши там, где хороши. Статья про то, где именно проходит граница.

Дисклеймер: текст вырос из внутреннего материала про сравнение нашего продукта с вайб-кодингом. Я постарался убрать рекламную часть и оставить только тезисы, которые применимы к любому no-code/low-code решению уровня Airtable/Notion/Baserow. Если они применимы — пишите в комментариях, дополним список.

## Сценарий: учёт заявок на сервисное обслуживание

Компания из 12 человек ведёт заявки на ремонт оборудования. Цепочка: диспетчер принимает звонок → инженер выезжает → бухгалтерия выставляет счёт. У каждой стороны свой объём прав, истории и интеграций с внешним миром.

На демо ИИ-инструмент за вечер выдаёт работающее веб-приложение:

- форма заявки;
- список со статусами;
- кнопки «Принять», «Передать инженеру», «Закрыть»;
- адаптивная вёрстка, тёмная тема, иконки, всё симпатично.

На презентации руководству — ровно та таблица, которую все ждали. Все довольны, договорились запускать со следующего понедельника.

### А потом наступает неделя внедрения

Спустя 3–5 дней команда задаёт вопросы, которых не было на демо:

1. **Где данные?** Сейчас они в IndexedDB/localStorage браузера? В SQLite на одном сервере без бэкапа? Если хост перезагрузится — мы их теряем?
2. **Резервная копия?** Кто, куда и с какой частотой её делает? Как восстановиться, если бухгалтерия удалила позавчерашние заявки?
3. **Кто что видит?** Диспетчер должен видеть все заявки, инженер — только свои, подрядчик — только заявки своей бригады. На бэкенде эти фильтры есть, или фильтрация в браузере? (Если в браузере — это не права доступа, это удобство интерфейса.)
4. **Что при одновременной правке?** Диспетчер и инженер открыли одну заявку и оба нажали «Сохранить». Чьё изменение перетёрло чьё?
5. **История изменений.** Кто и когда поменял исполнителя? Если поле просто перезаписано — этой информации нет.
6. **Экспорт.** Бухгалтерии раз в месяц нужна выгрузка в Excel с агрегатами. Это есть? Или копируем из таблицы вручную?
7. **Новый сотрудник.** Подключить ещё одного инженера: где это меняется? Прямо в коде? Если в коде — нужна доставка изменений на прод.
8. **Поддержка через год.** Автор сменил работу. Кто читает этот код? На каком фреймворке он написан? Зависимости обновляются?

Сгенерированный код по умолчанию не закрывает ни одного из этих вопросов. Каждый требует отдельной проработки: придумать схему данных, накатить миграции, добавить middleware прав, реализовать optimistic locking или транзакции, написать audit log, спроектировать форму выгрузки, отделить конфиг от кода, оформить деплой-пайплайн.

И вот тут «за вечер» превращается в проект на 2–3 месяца. Чаще всего его в итоге переписывают на классический стек — потому что ИИ-генерация рассчитана на быстрый старт, а не на эволюцию системы.

## Почему так получается

ИИ-инструменты вайб-кодинга оптимизированы под одну метрику: **первый экран, который виден через 5 минут**. Это правильно с точки зрения «вау-эффекта» и тестирования гипотезы. Проблема в том, что архитектурные решения, которые делают приложение учётной системой, **не видны** на первом экране и поэтому в генерации редко закладываются.

Сравните, что входит и не входит в типичный output вайб-кодинга:

| Свойство | Закладывается ИИ-генератором по умолчанию | Нужно проектировать отдельно |
|---|---|---|
| Внешний вид формы | ✅ | |
| Список с фильтрами/сортировкой | ✅ (фронтенд) | ✅ (фильтрация на сервере) |
| Поля и валидация фронта | ✅ | |
| Схема БД и миграции | частично | ✅ |
| Серверная авторизация и row-level security | | ✅ |
| Аудит изменений | | ✅ |
| Резервные копии | | ✅ |
| Конкурентные правки (locks, MVCC) | | ✅ |
| Экспорт в Excel/CSV/JSON | | ✅ |
| API для внешних интеграций | | ✅ |
| Обновления зависимостей | | ✅ |
| Документация для следующего разработчика | | ✅ |

Каждый «нужно проектировать отдельно» — это часы или дни инженерной работы. На 5 пунктах из таблицы экономия ИИ полностью съедается.

## Где правильный шов

Тезис не «ИИ-генератор бесполезен» — тезис другой: **ИИ хорошо генерирует интерфейс, плохо — основу данных**. Логичное разделение:

- **Данные, права, целостность, аудит, экспорт, API — на платформе** (Airtable, Notion, Baserow, NocoDB, Lark Base, любой нормальный no-code/low-code продукт с реляционной моделью и серверной авторизацией).
- **Кастомные интерфейсы, нестандартные дашборды, специфические рабочие места — генерируем ИИ** как HTML/JSX-шаблоны, подключённые к данным платформы через её API.

Тогда вайб-кодинг работает в режиме «гибкость заказного UI без проблем заказной разработки», а платформа закрывает всё остальное.

В нашем случае правильная схема оказалась такой (привожу как пример, не как рекламу — то же самое можно собрать на Baserow + custom frontend):

1. Данные хранятся в реляционной БД платформы. Бэкап и история изменений — штатные операции.
2. Роль «Подрядчик» имеет маску по полю «Исполнитель» — сервер отдаёт только записи, где исполнитель = текущий пользователь. Фильтр применяется ДО отдачи данных. Никакой ИИ-генерированный фронт не может обойти это, потому что отказ — на бэкенде.
3. Карточку заявки рисует HTML-шаблон, сгенерированный ИИ из описания «нужна карточка заявки с такими-то полями, тёмной темой и боковой панелью комментариев». Шаблон подключается к данным через отчёт/API платформы.
4. При попытке открыть чужую запись через тот же шаблон сервер просто отдаёт пустой набор — шаблон неважен.

Получается: эстетика ИИ-генератора + надёжность платформы. Без переписывания, потому что архитектурные вопросы решены не нами, а уровнем ниже.

Так вайб-кодинг не исчезает из процесса. Наоборот, он становится спокойнее: ИИ делает интерфейс поверх готового контура, а не притворяется всей системой.

## Где вайб-кодинг всё равно выигрывает

Не во всех задачах нужна учётная система. Если задача — **публичный лендинг**, **прототип-доказательство концепции для инвестора**, **внутренний экспериментальный инструмент на 2 человек**, **сложный графический интерфейс с тяжёлой клиентской анимацией** — вайб-кодинг даёт скорость, которую no-code-платформа не даст никогда.

Логика выбора:

- **Один экран без долгоиграющих данных** → вайб-кодинг.
- **Учётная система с правами, ростом команды, интеграциями** → платформа + ИИ только на UI.
- **Что-то посередине** → платформа, пока команда не упёрлась в её UI-ограничения, потом гибрид.

---

## Чек-лист «ИИ-генератор vs платформа учёта»

Когда команда предлагает «давайте просто сгенерируем ИИ-инструментом», стоит пройти этот короткий чек-лист. Если хотя бы 3 пункта отмечены — без платформы данных не обойтись.

- [ ] В системе будет более 1 роли пользователей с разным объёмом видимых данных.
- [ ] Записи редактируют параллельно более 1 человека.
- [ ] Нужна история изменений (кто, когда, что).
- [ ] Регулярная выгрузка во внешний формат (Excel, CSV, JSON).
- [ ] Интеграция с другими системами через API.
- [ ] Срок жизни системы — больше 6 месяцев.
- [ ] Расширение схемы данных без переписывания всей системы.
- [ ] Поддержка после ухода автора.
- [ ] Резервные копии и восстановление.
- [ ] Требования к хранению данных (ФЗ-152, КИИ, корпоративные политики).

## Что я вынес из этого сценария

Главное: вайб-кодинг лучше всего работает не как замена платформы, а как ускоритель интерфейсного слоя.

Для меня практическое правило теперь звучит так:

> **ИИ делает интерфейс. Платформа отвечает за данные, права и историю.**

Поэтому сейчас я сначала спрашиваю не «можно ли это сгенерировать», а другое:

- где будут храниться данные;
- кто и на каком уровне ограничивает доступ;
- как восстановиться после ошибки;
- как выгрузить данные наружу;
- кто сможет поддерживать решение без автора первого промпта.

Если ответы уже есть в платформе, ИИ можно безопасно пускать в интерфейсный слой. Если ответов нет, сгенерированный экран только маскирует будущую работу.

Это не аргумент против ИИ-инструментов. Я продолжаю использовать их там, где они дают быстрый выигрыш: структура страницы, HTML, CSS, сценарии отображения, черновики компонентов. Просто теперь я не называю готовым приложением то, что пока является только хорошим интерфейсом.

- Документация Baserow по правам уровня строк — хороший пример того, как должна выглядеть row-level security в no-code-продукте.

## Фактчек перед публикацией

- Подтвердить скриншотом или документацией, что ограничения доступа применяются на стороне сервера до отдачи данных в шаблон.
- Не привязывать выводы к конкретному AI-code инструменту без отдельного измеримого эксперимента.
- Подготовить скриншоты: таблица заявок, настройка роли подрядчика, HTML-шаблон карточки, шаблон в отчёте, попытка открыть чужие данные.
- Перед публикацией проверить, что canonical (`https://ideav.ru/knowledge-base/11-ai-interface-data-safety.html`) открывается, не отдаёт `noindex`, присутствует в `sitemap.xml` и доступен в `robots.txt` — иначе Habr-канонизация не сработает.
- На месте «нашего продукта» один раз в разделе «Где правильный шов» подставить «Интеграм» с canonical-ссылкой. Не больше одного упоминания во всём тексте.
- Перевод на английский для cross-post в dev.to, Medium, IndieHackers — через неделю после Хабра, не одновременно.
