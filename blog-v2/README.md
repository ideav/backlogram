# blog-v2 — скелет блога на `blog.ideav.ru`

Минимальный, рабочий скелет нового блога Интеграм на **Astro 5 + Tailwind 4**. Цель — заменить WordPress на `blog.ideav.online`. Новый блог — это самостоятельный канонический источник: каждая статья на `blog.ideav.ru` ссылается `<link rel="canonical">` сама на себя, а ссылка на оригинальную публикацию в старом блоге хранится отдельным полем `originalUrl` и показывается только как атрибуция «Первая публикация» (см. [issue #331](https://github.com/ideav/backlogram/issues/331)).

## Что уже есть

- Главная со списком статей: hero-карточка + сетка.
- Страница отдельной статьи (`/posts/<slug>/`) с author-блоком и breadcrumb.
- RSS-лента `/rss.xml`.
- `sitemap-index.xml` (генерируется автоматически через `@astrojs/sitemap`).
- SEO из коробки: canonical, OpenGraph, Twitter Card, JSON-LD-готовые метаданные, `robots.txt`.
- Тёмная тема через `prefers-color-scheme`.
- Три демо-статьи в `src/content/posts/` (одна новая + две из старого блога с атрибуцией `originalUrl`).
- Schema content collections с типизацией — Astro проверит frontmatter каждой статьи на этапе билда.

## Чего пока нет (намеренно)

- Постраничной разбивки списка (хватает на десятки статей, нужно при сотнях).
- Страниц категорий и тегов.
- Поиска по статьям (можно подключить Pagefind одной командой).
- Импорта 18 статей со старого WordPress — это отдельная задача (см. ниже).
- CI/CD — нет workflow для автодеплоя; всё локально.
- Картинок для соцсетей — заглушка `og-default.png` ещё не нарисована.

## Запуск локально

```bash
cd blog-v2
npm install
npm run dev   # http://localhost:4321
```

## Сборка для прода

```bash
cd blog-v2
npm install
npm run build   # результат в blog-v2/dist/
```

Контент `blog-v2/dist/` — статика. Кладётся на любой статический хостинг: Cloudflare Pages, Netlify, Vercel, Yandex Object Storage, обычный nginx — что удобнее.

## Как добавить статью

Создать файл `src/content/posts/<slug>.md`:

```markdown
---
title: "Заголовок статьи"
description: "Один абзац для превью и meta description (150–160 символов)."
pubDate: 2026-05-20
category: "Технологии"
author: "Имя автора"
image: "https://example.com/cover.jpg"   # опционально
draft: false                              # true — не попадёт в билд
originalUrl: "https://..."                # опционально: URL первой публикации (атрибуция, не влияет на canonical)
canonical: "https://..."                  # опционально: SEO-canonical override; ставить ТОЛЬКО если оригинал на другом сайте — авторитетный источник
---

## Заголовок раздела

Текст статьи в обычном Markdown.
```

`<slug>` — это URL: статья `src/content/posts/moya-statya.md` будет доступна по `/posts/moya-statya/`.

## Импорт статей со старого блога

**Сделан.** В `src/content/posts/` лежит 50 статей: 49 импортировано с `blog.ideav.online` + одна welcome-статья.

Импорт сделан Python-скриптом, который:

1. Тянет URL всех статей через `sitemap.xml` + дополняет тремя свежими статьями из RSS (sitemap WP не всегда обновляется).
2. Скачивает HTML каждой страницы.
3. Берёт `<div itemprop="articleBody">` как основное тело, плюс title, дату публикации, категорию, автора из микроразметки.
4. Конвертирует HTML → Markdown через `html2text` (тела таблиц сохранены как pipe-таблицы).
5. Сохраняет URL первой публикации в поле `originalUrl` (атрибуция «Первая публикация»). Раньше этот URL подставлялся в `canonical`, из-за чего `blog.ideav.ru` указывал каноничным адресом старый блог и Яндекс исключал страницы как «Малоценная или маловостребованная страница» — см. [issue #331](https://github.com/ideav/backlogram/issues/331). Теперь каждая статья самоканонична на `blog.ideav.ru`.
6. Описания, обрезанные на полуслове в WP meta-description, заменены первой осмысленной фразой из тела.

### Что осталось не идеально

- **Категории как есть со старого блога** (10 разных значений: «Без категории», «Технологии», «О платформе», «Обучение», «Проекты», «Разработка», «Правила предпринимателя в IT» и т.д.). Возможно, стоит причесать вручную.
- **Изображения остались с ссылками на старый блог** (`blog.ideav.online/wp-content/uploads/...`). Они работают, но при отказе старого хостинга пропадут. Перед запуском желательно скачать `public/uploads/` локально и переписать пути.
- **Внутренние ссылки между статьями** ведут на `blog.ideav.online/...`. После запуска нового блога стоит переписать на относительные `/posts/<slug>/`.
- **Несколько статей с короткой шапкой** (одно-два предложения) — те, что были анонсами или короткими записями вроде поздравления с НГ. Не критично.
- **Авторы** все приведены к «Команда Интеграм» — в WP они шли как `Integram`/`admini`.

## Деплой на `blog.ideav.ru`

DNS уже не указывает на старый блог (тот живёт на `blog.ideav.online` отдельно), поэтому переезд несложный:

1. Залить `dist/` на выбранный хостинг.
2. В DNS добавить запись `blog.ideav.ru → CNAME → <хостинг>`.
3. Прописать SSL (Let's Encrypt автоматический на большинстве хостингов).
4. Проверить:
   - `https://blog.ideav.ru/` — главная грузится;
   - `https://blog.ideav.ru/rss.xml` — RSS работает;
   - `https://blog.ideav.ru/sitemap-index.xml` — sitemap есть;
   - `https://blog.ideav.ru/robots.txt` — корректный;
   - в `<head>` страницы статьи есть `<link rel="canonical">`, OG-теги, JSON-LD.

## Что отличается от старого `blog/` (PR #73)

Папка `blog/` в корне репо — это **React-SPA**, который читает RSS со старого WordPress и показывает превью-карточки со ссылкой обратно на WP. Это другая архитектура и другая задача: витрина поверх существующего WP.

`blog-v2/` — это **самостоятельный блог**, который хранит свой контент в репо и не зависит от WordPress в проде. После переезда `blog/` либо удалится, либо превратится в админку.
